<!-- FILL:surfaces -->
- **app:** model-driven Customer Service app for service representatives and supervisors, with Copilot Service admin center used only for configuration tasks.
- **case form:** show customer, Priority, current status, `Is Escalated`, and SLA timer controls for first response / resolution on the main Case form so the operator sees tier and time pressure in one place.
- **views:**
  - representative views for active cases by priority/tier;
  - supervisor views for escalated active cases and breached-but-unassigned cases;
  - queue views for the supervisor escalation queue.
- **dashboards:** supervisor dashboard tiles/charts for escalated backlog, breaches by tier, and paused/on-hold SLA work.
- **layout:** keep the experience in Unified Interface OOB grids/forms/dashboards so the layout reflows responsively and uses the platform's existing site-map navigation.

Grounding:
- https://learn.microsoft.com/power-apps/user/unified-interface
- https://learn.microsoft.com/dynamics365/customer-service/administer/define-service-level-agreements
<!-- /FILL -->

<!-- FILL:client-logic -->
Business-rule first, then low-code, with no custom script:
- server-side logic owns tier-to-priority stamping and SLA breach escalation so the outcome is consistent for manual and automated case-creation paths;
- form-level logic is limited to presentation, such as showing timers and making escalation indicators visible/read-only to the right personas;
- no JavaScript, PCF, or custom page logic is introduced because the approved feature only needs standard form/view/dashboard behavior plus server-side SLA automation.

Grounding:
- https://learn.microsoft.com/dynamics365/customer-service/administer/automatically-create-update-records
- https://learn.microsoft.com/dynamics365/customer-service/administer/create-enhanced-sla
<!-- /FILL -->

<!-- FILL:components -->
- **OOB main Case form** — justified because the feature needs standard case fields, SLA timer controls, and queue actions already supported by model-driven forms.
- **OOB views and dashboards** — justified because escalated work monitoring is list/chart based, not a bespoke interaction model.
- **OOB queue dialogs and queue grids** — justified because supervisors manage escalated work through queue items already supported in Customer Service.
- **No PCF / no custom page** — justified because Microsoft already documents accessible Unified Interface grids/forms/dashboards and the approved scope adds no novel visualization requirement.

Grounding:
- https://learn.microsoft.com/power-apps/user/unified-interface
- https://learn.microsoft.com/power-apps/user/screen-reader
- https://learn.microsoft.com/dynamics365/customer-service/administer/set-up-queues-manage-activities-cases
<!-- /FILL -->

<!-- FILL:navigation-accessibility -->
- Keep FEAT-02 inside the standard model-driven site map so keyboard users can reach cases, queues, and dashboards through the normal navigation structure.
- Preserve a predictable tab order on the Case form and avoid custom controls that would bypass the documented screen-reader support for grids, forms, dashboards, and dialogs.
- Use field labels and dashboard/view names that match the approved business language (`Gold`, `Silver`, `Bronze`, `Is Escalated`, `Supervisor Escalations`) so Narrator/JAWS/NVDA announcements stay understandable.
- Rely on Unified Interface responsive reflow rather than custom layout logic so smaller screens still expose timers and escalation indicators without horizontal-navigation traps.

Grounding:
- https://learn.microsoft.com/power-apps/user/screen-reader
- https://learn.microsoft.com/power-apps/user/unified-interface
<!-- /FILL -->
