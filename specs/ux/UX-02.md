<!-- FILL:surfaces -->
- **app:** model-driven Customer Service app for service agents and supervisors; Copilot Service admin center used only for SLA, queue, and field-security-profile configuration tasks.
- **Case main form:** a new Escalation section (shown only when Case_Escalated=true via a server-side business rule) contains the Escalated indicator (read-only Boolean) and the EscalationNotes field (visible only to users assigned the EscalationNotes FieldSecurityProfile — Supervisor and Manager roles). This section is appended below the existing form layout so it does not disrupt the established SLA timer controls and case fields introduced by FEAT-02.
- **Active escalations view:** model-driven system view on the Case table, filtered to Case_Escalated=true AND Status=Active, sorted ascending by remaining SLA breach time (most urgent at the top). Columns: Case Number, Customer, Priority, Remaining Time to Breach. Accessible from the Customer Service app site map and embedded in the supervisor dashboard (REQ-0007).
- **Supervisor escalations dashboard:** model-driven system dashboard embedding the active escalations view and a chart showing escalation urgency distribution (grouped by remaining-time buckets). Accessible to Supervisor and Manager roles from the Customer Service app; provides the real-time escalation backlog visibility required by REQ-0007.
- **Layout:** all surfaces use the Unified Interface OOB grid/form/dashboard layout with responsive reflow; no PCF component, canvas page, or custom HTML surface is introduced.

Grounding:

<!-- FILL:client-logic -->
Business rule first, then server-side low-code, with no client-side JavaScript:

- The Case_Escalated flag is set exclusively by the server-side escalation trigger flow (invoked by the SLA warning action), ensuring consistent escalation state regardless of how the case is accessed (form, API, bulk operation).
- The EscalationNotes field visibility is enforced at the Dataverse server layer via the EscalationNotes FieldSecurityProfile; the column is not returned in API responses for users without the profile, providing defence-in-depth that does not rely on client-side hiding.
- The Escalation section on the Case form is shown conditionally via a model-driven server-side Business Rule (no JavaScript): condition `Case_Escalated = true` → show Escalation section; `Case_Escalated = false` → hide Escalation section. This keeps the form uncluttered for non-escalated cases without introducing any client-side script dependency.
- No JavaScript, PCF, or custom page logic is introduced for FEAT-03; the approved requirements only require standard field display, read-only indicators, a sorted view, and a dashboard — all achievable with OOB model-driven capabilities.

Grounding:
- https://learn.microsoft.com/power-apps/maker/model-driven-apps/define-data-model-driven-app

<!-- FILL:components -->
- **OOB Case main form (Escalation section update)** — justified: the existing Case main form is extended with a single new section containing two fields (Boolean indicator + secured text field); no novel interaction model or PCF rendering is required. Reusing the OOB form avoids introducing a separate custom page and keeps the escalation context co-located with the case record supervisors already work from.
- **OOB Active escalations view** — justified: a model-driven system view with filter criteria (Escalated=true, Status=Active) and ascending sort by remaining breach time fully satisfies REQ-0007's requirement for an urgency-ordered list. No bespoke chart rendering or custom pagination is needed.
- **OOB Supervisor escalations dashboard** — justified: a model-driven system dashboard embedding the view and a standard chart covers the supervisor operational visibility requirement without a Power BI embedded report or canvas app. Microsoft Learn documents system dashboards as a supported low-code surface for role-based operational monitoring.
- **OOB server-side Business Rule (section visibility)** — justified: conditional section visibility based on a Boolean field value is a standard use case for server-side model-driven Business Rules; no JavaScript required, ensuring the logic runs consistently across web, mobile, and API access paths.
- **No PCF / no custom page** — justified: no novel visualization, custom input interaction, or data rendering beyond OOB model-driven capabilities is required by the approved FEAT-03 scope. Avoiding PCF and custom pages minimises ALM complexity and reduces the surface area for accessibility regressions.

<!-- FILL:navigation-accessibility -->
- **Navigation:** the supervisor escalations dashboard and active escalations view are added to the Customer Service app site map under a dedicated "Escalations" group (or within the existing "Cases" area). This ensures keyboard users can reach the dashboard and view through the standard Unified Interface site map navigation without custom hot keys or workarounds.
- **Screen reader labels:** the Escalation section heading ("Escalation"), field labels ("Escalated", "Escalation Notes"), and view column headers ("Case Number", "Customer", "Priority", "Remaining Time to Breach") use plain descriptive text. Narrator, JAWS, and NVDA can announce all values via the standard model-driven ARIA attributes generated by Unified Interface.
- **Field security and tab order:** the EscalationNotes field is absent from the rendered DOM for users without the EscalationNotes FieldSecurityProfile (the column is not returned by the Dataverse API). There are no hidden but focusable elements or empty tab stops that would cause screen-reader confusion.
- **Conditional section:** the Escalation section is hidden for non-escalated cases via the server-side Business Rule. When hidden, the section is not rendered in the DOM, so there is no ARIA-hidden pitfall for assistive technology.
- **Responsive reflow:** the Unified Interface OOB dashboard and grid provide responsive reflow for smaller viewports; no horizontal-navigation trap or fixed-width layout is introduced.
- **Colour contrast:** the Escalated indicator uses the OOB Boolean field control (true/false text or checkbox) rather than a custom colour-coded badge, avoiding colour-only communication and WCAG contrast failures.

Grounding:
